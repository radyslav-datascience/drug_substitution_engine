## What this is

A pharmacy chain loses revenue every time a customer walks out without a prescribed drug. The strategic question is **not** "how often are we out of stock" — that is operations. The question is: **when we are out, do customers buy a substitute from us, or do they leave?**

This engine answers that question for **6 264 drugs** across **150+ local markets** (OPT-codes ≈ city/district granularity), using a difference-in-differences (DiD) design on stockout events. Every drug gets:

- `COEF_1` — average share of demand the chain retains when the drug is out (after IQR-trim across markets).
- `COVERAGE_PCT × CONDITIONAL_RETENTION` — the decomposition that explains *why* `COEF_1` is what it is.
- `RELIABILITY_SCORE` — composite confidence in `COEF_1` (CV-based stability × log-saturated sample size × modality penalty).
- `DRUG_CLASS` — `UNIMODAL` / `MULTIMODAL` (Hartigan's dip test) flagging drugs whose mean is statistically misleading.
- 16 validation invariants, all green.

The output feeds a tier-based stocking strategy (A: critical / B: standard / C: watch) and a Power BI workbook the analyst can open without further work.

## Quick start

```bash
# Windows — one-click run
run.bat

# Or from CLI
python -m pipeline.full_run                       # full run, 20-market threshold
python -m pipeline.full_run --limit 5             # smoke-test on 5 markets
python -m pipeline.full_run --workers 4           # 4 workers instead of 6
python -m pipeline.full_run --force               # ignore parquet cache
python -m pipeline.full_run --min-market-count 3  # softer threshold
```

A run on the reference dataset (≈30 GB raw CSV) takes **~45 minutes** on 6 cores / 32 GB RAM. The pipeline is **resume-aware**: every per-market parquet is corruption-checked on load, and an interrupted run picks up where it stopped.

The engine is dataset-agnostic. To run on your own pharmacy data, drop transactional CSVs into `data/raw/` matching the schema in [`docs/_data_format.md`](docs/_data_format.md). NFC1 reference data is loaded from `data/master/nfc1_config.json`.

## Pipeline

```
Raw CSV (data/raw/*.csv)
   │
   ▼
Phase A0: Discover markets ─────────────────► markets_list.csv
   │
   ▼
Phase A: Per-market processing  (parallel × N workers)
   ├─ A1  Data aggregation          ─► aggregated.parquet
   ├─ A2  Stockout detection        ─► stockout_events.parquet
   ├─ A3  DiD analysis              ─► did_events.parquet + substitute_pairs.parquet
   └─ A4  Substitute share analysis ─► substitute_shares.parquet
   │
   ▼
Phase B: Cross-market aggregation ──────────► drug_statistics.parquet
   │                                          (broad: ALL drugs, full visibility)
   ▼
Phase C: Final export ──────────────────────► 4 customer-facing files
                                              (narrow: filters phantoms,
                                               applies invariants)
```

The "broad model, narrow export" pattern is deliberate: Phase B keeps every intermediate row so debugging stays cheap, while Phase C is the only place client-facing filters live. A bug in a downstream filter never silently corrupts upstream math.

A live Rich-based TUI dashboard shows per-market progress, ETA, and validation invariants during the run.

## Key formulas

```
SHARE_INTERNAL_m = INTERNAL_LIFT_m / (INTERNAL_LIFT_m + LOST_SALES_m)
                                                              ◄ per market m

COEF_1           = mean( IQR_trim_1.5( {SHARE_INTERNAL_m | m ∈ markets} ) )
                                                              ◄ across markets

COEF_1           = COVERAGE_PCT × CONDITIONAL_RETENTION       ◄ decomposition
   COVERAGE_PCT  = | { m : SHARE_INTERNAL_m > 0 } | / | markets |
   CONDITIONAL_RETENTION = mean( SHARE_INTERNAL_m | SHARE_INTERNAL_m > 0 )

RELIABILITY_SCORE = stability × sample_factor × modality_penalty
   stability         = 1 − CV(SHARE_INTERNAL_m)               ∈ [0, 1]
   sample_factor     = log10(1 + n_markets) / log10(1 + 150)  saturates at n=150
   modality_penalty  = 0.85  if  DRUG_CLASS = MULTIMODAL  else 1.00

DRUG_CLASS         = MULTIMODAL  if Hartigan dip test p < 0.05  else UNIMODAL
```

Why the mean (not the median) on `SHARE_INTERNAL`? With IQR-trim already removing the long tails, the mean preserves more signal at the high end of the distribution where retention concentrates. `_methods_issues.md::ISSUE-013` walks through the change.

Why decomposition? A single `COEF_1 = 0.50` is ambiguous (see scatter chart above). Giving the operator both `COVERAGE_PCT` and `CONDITIONAL_RETENTION` makes that ambiguity visible — and is essential for `MULTIMODAL` drugs where the mean lies between two real modes.

Why a composite reliability? Per-drug confidence depends on three independent things: how stable the share is across markets (`stability`), how many markets the drug appeared in (`sample_factor`), and whether the share distribution has one mode or several (`modality_penalty`). Wrapping them into a single `[0, 1]` score lets the analyst sort and threshold without juggling three fields.

## Architecture principles

1. **Pipeline vs ad-hoc are physically separated.** `pipeline/` is locked-in production code — touched only with a methodology issue logged in `docs/_methods_issues.md`. Ad-hoc analytics live in `_optional_calculations/<task>/` with their own `config.py`, `run_*.py`, `inputs/`, `outputs/`, `logs/`. They depend on the pipeline; the pipeline never depends on them.

2. **Broad model, narrow export.** Phase B keeps every drug — phantom substitutes, low-coverage rows, everything. Phase C is the only place the customer-facing filter (`SHARE > 0`, `MARKET_COUNT ≥ N`, etc.) lives. Debugging stays cheap; production stays clean.

3. **Math first, code second.** Every coefficient has a formula in `docs/ALGORITHMS.md`, a validation invariant in `validation_report.txt`, and a row in `data_dictionary.txt`. If those three disagree, the engine refuses to publish.

4. **Resume-aware, corruption-checked.** Per-market parquets are revalidated on load; a partially-written cache is detected and rebuilt. A 45-minute run can survive a kernel restart.

5. **Single source of truth for NFC1.** `data/master/nfc1_config.json` is the only place the NFC1 family registry lives. Pipeline, ad-hoc tasks, and reports all read it.

## Data privacy

This repository contains **code, methodology, and aggregated/anonymized reports only**. Customer data is excluded by `.gitignore` and never reaches the public history. Specifically:

- ❌ `data/raw/`, `data/intermediate/`, `data/master/*.json` — customer transactional data and NFC1 configuration.
- ❌ `results/final/*` — drug-level output files (kept locally; folder structure preserved with `.gitkeep`).
- ❌ `_optional_calculations/*/inputs/`, `_optional_calculations/*/outputs/` — analyst-facing intermediates.
- ❌ `logs/` — runtime logs may contain real `CLIENT_ID`s.
- ✅ Drug names, INN, NFC1 — kept (public pharma reference, identical across vendors).
- ✅ Reports in `reports/` — explicitly anonymized; the customer is referred to as "the pharmacy chain."

Full details and the responsible-disclosure address: [`SECURITY.md`](SECURITY.md).

## Tech stack

- **Python 3.13** — `pandas`, `numpy`, `pyarrow`, `scipy.stats` (Hartigan dip test), `openpyxl`, `rich` (TUI).
